### 02. Webpack과 Bundle

---

> Webpack은 여러 개로 나뉘어 있는 프론트엔드 리소스(JavaScript, CSS, 이미지 등)를 하나 또는 여러 개의 번들(Bundle) 파일로 묶어주는 모듈 번들러이다.

<br>

#### 1. 번들(Bundle)이 필요한 이유

---

- 과거에는 HTML 파일에서 필요한 리소스를 각각 직접 불러오는 방식이 일반적이었다.

```html
<script src="a.js"></script>
<script src="b.js"></script>
<script src="c.js"></script>
```

- 하지만 애플리케이션 규모가 커질수록 다음과 같은 문제가 발생한다.
  - JavaScript 파일 개수가 많아짐
  - 네트워크 요청 수 증가 → 초기 로딩 속도 저하
  - 파일 간 의존성 관리가 어려움
  - 브라우저 환경별 최적화가 어려움
- 이 문제를 해결하기 위해 등장한 개념이 **번들링(Bundle)** 이다.

<br>

#### 2. 번들이란?

---

- **번들(Bundle)** 이란 여러 개의 모듈 파일을 하나(또는 여러 개)의 파일로 묶은 결과물이다.
- 브라우저는 최종적으로 번들 파일만 다운로드하고 실행한다.

```
src/
 ├─ index.js
 ├─ api.js
 ├─ utils.js
 └─ component.js

↓

 dist/
 └─ bundle.js

```

<br>

#### 3. Webpack의 역할

---

- Webpack은 단순히 파일을 합치는 도구가 아니다. **애플리케이션의 진입점부터 모든 의존성을 분석해 번들을 생성한다.**
- Webpack이 하는 주요 역할
  - Entry
    - 애플리케이션의 시작점 지정
    - 예: index.js
  - Dependency Graph 생성
    - import, require를 따라가며 모든 모듈의 의존 관계를 그래프로 구성
  - Bundle 생성
    - 의존성 그래프를 기반으로 하나 이상의 번들 파일 생성
  - Loader 처리
    - JavaScript 외의 리소스를 처리 가능하게 변환
    - 예:
      - babel-loader → 최신 JS → 브라우저 호환 JS
      - css-loader → CSS를 JS 모듈로 변환
      - file-loader → 이미지, 폰트 처리
  - Plugin 처리
    - 번들 최적화, 파일 분리, 환경 변수 주입 등 추가 작업 수행

<br>

#### 4. Webpack 번들과 브라우저 렌더링의 관계

---

- Webpack은 브라우저가 코드를 실행하기 전에 필요한 준비 과정을 담당한다.
- 브라우저 관점에서 보면:
  - 서버 응답 →
  - HTML 파싱 →
  - `<script src="bundle.js">` →
  - JS 파싱 및 실행
- 즉, 브라우저는 Webpack을 알지 못한다. 브라우저는 단지 bundle.js라는 하나의 JS 파일을 다운로드하고 실행할 뿐이다.

<br>

#### 5. 번들이 렌더링 성능에 미치는 영향

---

- 번들 전략은 렌더링 성능에 직접적인 영향을 준다.
- 하나의 거대한 번들 (X)
  - 초기 다운로드 용량 증가
  - JS 파싱 & 실행 시간 증가
  - LCP, TTI 성능 악화
- 적절히 분리된 번들(Code Splitting) (O)
  - 초기 로딩 시 필요한 코드만 다운로드
  - 나머지는 필요할 때 로딩
  - 초기 렌더링 속도 개선
- Webpack은 번들을 여러 개로 나누는 코드 스플리팅을 지원한다.
  - 초기 렌더링 시 필요한 코드만 먼저 로드
  - 이후 사용자 인터랙션 시 추가 번들 로드

<br>

#### 6. 정리

---

- Webpack은 프론트엔드 리소스를 **브라우저가 이해하고 실행하기 쉬운 형태로 묶는 도구**
- 번들은 브라우저가 실제로 다운로드하고 실행하는 **최종 산출물**
- 번들 크기와 구조는 **렌더링 성능에 직접적인 영향을 준다**
